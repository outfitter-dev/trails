# ADR-0018: Signal-Driven Governance

## Context

The framework's operational lifecycle is a collection of separate subsystems with separate execution models:

- **The warden** runs 11 AST-based TypeScript rules by iterating the topo. Load source files, parse ASTs, traverse the topo Map, emit diagnostics.
- **The lockfile** is generated by `writeSurfaceLock()` during a build step. Serialize the topo to JSON, hash it, write the file.
- **The surface map** is generated by `deriveSurfaceMap()`. Iterate the topo, derive JSON Schemas, sort, serialize.
- **The tracing** records execution data through a layer. Nothing reads it back in a useful way.
- **Schema cache** (proposed in ADR: Topo Store) checks a hash, computes if stale, stores.

Each subsystem has its own trigger mechanism, its own execution model, its own error handling, and its own testing story. None of them are trails. None of them benefit from the framework's own primitives: typed schemas, examples, `testAll`, governance, crossings, resources.

This is the cobbler's children problem. The framework that exists to eliminate redundant authoring has redundant, hand-written operational subsystems.

### The insight

Warden rules are already trails (`wardenTopo` exists in the current codebase). But they run in their own isolated topo with their own execution path. The lockfile generator is a pure function from topo state to text output. The surface map generator is a pure function from topo state to JSON. The tracing aggregation queries are pure functions from execution records to summaries.

These are all trails. They take input (topo state, execution records, source files), produce output (diagnostics, lockfiles, reports), and have testable examples. The only reason they're not expressed as trails is that the framework's operational lifecycle predates the primitives that could express it.

With the topo store (ADR: Topo Store) providing a queryable structural graph as a read-only resource, and signals providing typed activation, the entire operational lifecycle becomes trails that fire on signals.

## Decision

### Framework signals

Three signals define the framework's operational lifecycle:

```typescript
const topoSaved = signal('topo.saved', {
  payload: z.object({
    snapshotId: z.string(),
    previousSnapshotId: z.string().optional(),
    changedTrails: z.array(z.string()),
    addedTrails: z.array(z.string()),
    removedTrails: z.array(z.string()),
  }),
});

const topoPinned = signal('topo.pinned', {
  payload: z.object({
    snapshotId: z.string(),
    pin: z.string(),
    gitSha: z.string().optional(),
  }),
});

const trackRecorded = signal('track.recorded', {
  payload: z.object({
    trackId: z.string(),
    trailId: z.string(),
    status: z.enum(['ok', 'err']),
    durationMs: z.number(),
  }),
});
```

`topo.saved` is emitted whenever the framework writes a new topo snapshot. The signal retains its original `topo.saved` name for continuity with existing subscribers. `topo.pinned` is emitted when a developer deliberately retains the current topo under a durable pin name. `track.recorded` comes from the tracing layer. Together they drive governance, lockfile export, and dev intelligence trails.

### The lockfile as a trail

The lockfile generator becomes a trail that fires on topo snapshots and pins:

```typescript
trail('framework.lock.export', {
  on: [topoSaved],
  resources: [topoStore],
  intent: 'write',
  blaze: async (input, ctx) => {
    const conn = topoStore.from(ctx);
    const snapshot = await conn.snapshots.get({ snapshotId: input.snapshotId });
    const trails = await conn.trails.list({}, { orderBy: 'id' });
    const crossings = await conn.crossings.list({});
    const lockContent = renderLock(snapshot, trails, crossings);
    await writeFile('.trails/trails.lock', lockContent);
    return Result.ok({ path: '.trails/trails.lock', trailCount: trails.length });
  },
  examples: [
    {
      name: 'Empty topo produces minimal lockfile',
      input: { snapshotId: 'test-snapshot', changedTrails: [], addedTrails: [], removedTrails: [] },
    },
  ],
});
```

The lockfile is a projection of a signal. Same execution model as any other trail. Testable. Governable. Has examples.

**Dev vs CI behavior.** In CI, this trail fires on every `topo.saved` signal. In dev, it fires only on `topo.pinned` — explicit developer landmarks, not every file-save refresh. The CI topo includes the lockfile trail with `on: [topoSaved]`. The dev topo includes it with `on: [topoPinned]`. This is configuration, not conditional logic.

### Warden rules as trails

Warden rules evolve from standalone TypeScript functions to trails that declare the topo store as a resource. The two categories map cleanly:

**Structural rules** (query the topo store):

```typescript
trail('warden.write-needs-output', {
  resources: [topoStore],
  intent: 'read',
  description: 'Write-intent trails must declare an output schema',
  examples: [
    {
      name: 'All write trails have output',
      input: {},
      expected: { pass: true, diagnostics: [] },
    },
  ],
  blaze: async (_input, ctx) => {
    const conn = topoStore.from(ctx);
    const writeTrails = await conn.trails.list({ intent: 'write' });
    const missing = writeTrails.filter(t => !t.hasOutput);
    return Result.ok({
      pass: missing.length === 0,
      diagnostics: missing.map(t => ({
        trailId: t.id,
        message: 'Write-intent trail missing output schema',
        severity: 'error',
      })),
    });
  },
});
```

**AST rules** (inspect source code) keep their current approach but are expressed as trails with source file input:

```typescript
trail('warden.no-process-exit', {
  intent: 'read',
  input: z.object({
    filePath: z.string(),
    sourceCode: z.string(),
  }),
  blaze: async (input, ctx) => {
    // AST inspection, same as today but in trail form
  },
});
```

Both kinds run through the same execution pipeline, produce the same diagnostic format, and are testable with examples.

### Dev intelligence as trails

Dev intelligence queries become trails that fire on `track.recorded` or run on demand:

```typescript
trail('dev.latency-report', {
  resources: [topoStore],
  intent: 'read',
  input: z.object({
    trailId: z.string().optional(),
    limit: z.number().default(20),
  }),
  blaze: async (input, ctx) => {
    const conn = topoStore.from(ctx);
    const report = await conn.query(sql`
      SELECT t.id, t.intent,
        COUNT(*) as invocations,
        ROUND(AVG(tr.ended_at - tr.started_at), 1) as avg_ms,
        MAX(tr.ended_at - tr.started_at) as max_ms
      FROM topo_trails t
      JOIN track_records tr ON t.id = tr.trail_id
      WHERE t.snapshot_id = ${currentSnapshotId}
        AND tr.ended_at IS NOT NULL
        ${input.trailId ? sql`AND t.id = ${input.trailId}` : sql``}
      GROUP BY t.id ORDER BY avg_ms DESC
      LIMIT ${input.limit}
    `);
    return Result.ok({ entries: report });
  },
});
```

These are exposed via `trails dev stats`, `trails dev errors`, `trails dev trace <id>`. Because they're trails, they get MCP tool exposure for free. An agent calls `dev_latency_report` to understand the app's runtime characteristics.

### The framework topo

All framework-internal trails (lockfile, warden rules, dev intelligence) are assembled into a framework topo:

```typescript
import * as lockModule from './trails/lock';
import * as wardenModule from './trails/warden';
import * as devModule from './trails/dev';

export const frameworkTopo = topo('trails',
  lockModule,
  wardenModule,
  devModule,
);
```

This topo is:

- **Governable by itself.** The warden can check warden rules. A meta-warden rule can verify that all warden rules have examples.
- **Testable.** `testAll(frameworkTopo)` verifies every governance rule, every lockfile edge case, every dev intelligence query. The topo store's mock resource seeds test data.
- **Introspectable.** `trails topo` shows the framework's own operational topology summary, and `trails survey <id>` plus role-specific survey trails inspect details. What fires on what. What resources are needed. The full picture.

### Signal scoping

Not every framework trail fires in every environment. The lockfile trail fires on every topo snapshot in CI but only on pins in dev. Dev intelligence trails are irrelevant in CI. Some warden rules only apply during development.

Different framework topos per environment handle this cleanly:

```typescript
// CI: full governance + lockfile on every topo snapshot
const ciFrameworkTopo = topo('trails', lockCiModule, wardenModule);

// Dev: intelligence + lockfile on pin only
const devFrameworkTopo = topo('trails', lockDevModule, wardenModule, devModule);
```

The trails are environment-agnostic. The topo composition determines what's active. This follows the existing pattern: app developers already compose different topos for different deployment targets.

## Consequences

### Positive

- **The framework eats its own cooking.** Governance, lockfile generation, and dev intelligence use the same primitives — trails, resources, signals, fires — that app developers use. Every improvement to those primitives benefits the framework itself.
- **Everything is testable.** Warden rules have examples. The lockfile generator has examples. Dev intelligence queries have examples. `testAll(frameworkTopo)` verifies all of it.
- **Everything is governable.** The warden governs itself. Meta-governance — do all warden rules have examples? do all framework trails declare intent? — falls out naturally.
- **Signals decouple producers from consumers.** The build system emits `topo.saved`. `trails topo pin` emits `topo.pinned`. Neither knows or cares what fires on them. New governance rules, reports, and integrations are trails that fire on existing signals.
- **Dev intelligence joins structure and behavior.** The topo store (structure) and tracing (behavior) are both resources. Trails that cross-query them produce insights neither could generate alone.

### Tradeoffs

- **Bootstrap complexity.** The framework topo needs the topo store resource, which needs the core database, which is populated by the build system. Initialization order must be carefully managed — the database must be populated before the framework topo's trails can fire.
- **Two topos in one process.** In dev mode, the app topo and the framework topo coexist. They share `trails.db` but have separate trail registries. Clear namespacing is required (framework trails use the `framework.*` or `warden.*` prefix).
- **Incremental migration.** The existing warden package has 11 working rules. Migrating them to trail form is incremental — both standalone functions and trail-form rules coexist during the transition.

### What this does NOT decide

- **The complete set of framework signals.** Additional signals emerge as other subsystems adopt this pattern.
- **Specific warden rule implementations.** Each rule migrates independently.
- **Whether the framework topo is visible to app developers or strictly internal.**
- **Notification and webhook integrations.** A trail that fires on `topo.pinned` and posts to Slack is natural but out of scope.
