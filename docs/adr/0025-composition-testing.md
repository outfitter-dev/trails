# ADR-0025: Composition Testing

## Context

### Examples work brilliantly for single trails

The examples-as-tests system is the framework's crown jewel. One `testAll(app)` call runs every example as an assertion. The developer writes structured data on the trail definition and gets six consumers for free: testing, agent guidance, schema validation, mock data, composition testing, and contract coverage[^1].

For single trails, this works perfectly. The trail receives input, runs the blaze, returns a Result. The example declares input and expected output. The assertion is a deep equality check or schema validation.

### Composite trails break the assertion model

When a trail crosses other trails, its output depends on the crossed trail's behavior, which depends on the state those trails see. In the Stash dogfood app, `gist.fork` crosses `gist.show` and `gist.create`. Its output includes fields generated by the downstream trail — IDs, timestamps, computed values. The example can't predict these because they're runtime-derived.

Today the developer has two options, neither good:

1. **Full `expected` match.** Hard-code the exact output, including generated fields. This couples the example to implementation details of downstream trails and breaks when those trails change.
2. **Schema-only match.** Omit `expected` entirely. The test verifies the output is valid against the schema but doesn't assert anything about the actual values. For a composite trail where the interesting behavior IS the composition — what got passed through, what got transformed, what got combined — schema-only tells you almost nothing.

The gap: there's no way to say "the output should contain these specific fields with these values, and I don't care about the rest."

### Multi-step flows have no testing primitive

A single trail example tests one input/output pair. But many meaningful behaviors span multiple trail invocations in sequence. An onboarding flow: create user, assign role, send welcome notification. A fork flow: show original, create copy, record relationship.

These flows are testable today by writing manual test code with `createCrossContext` and `createTestContext`. But they're not expressible as structured data — which means they don't feed the six consumers. An agent reading examples sees individual trail behaviors but can't see how trails compose into journeys. The testing guide can't generate flow documentation. The warden can't check flow coverage.

## Decision

### `expectedMatch` for partial assertions on examples

A new field on trail examples, alongside `expected` and `error`. It asserts that the output is a superset of the declared shape — matching the specified fields exactly while ignoring fields not mentioned:

```typescript
trail('gist.fork', {
  crosses: [showGist, createGist],
  examples: [
    {
      name: 'Fork a gist',
      input: { id: 'g1' },
      expectedMatch: {
        description: 'Hello world',
        content: '# Hello',
        forkedFrom: 'g1',
      },
      // id, createdAt, etc. are NOT asserted — they're generated
    },
  ],
  // ...
});
```

**The assertion hierarchy is now three levels:**

| Field | Assertion | When to use |
|---|---|---|
| `expected` | Deep equality — output must match exactly | Simple trails with fully predictable output |
| `expectedMatch` | Partial match — declared fields must match, others ignored | Composite trails with generated or unpredictable fields |
| *(neither)* | Schema-only — output must be valid against output schema | Smoke tests, exploratory examples |

All three remain structured data on the trail definition. All three feed the same six consumers. `expectedMatch` fills the gap between "I know everything" and "I know nothing."

### The matching semantics

`expectedMatch` uses subset matching:

- **Scalar values** — strict equality.
- **Objects** — every key in `expectedMatch` must exist in the output with a matching value. Extra keys in the output are ignored.
- **Arrays** — by default, the output array must contain every element in the `expectedMatch` array (order-independent, subset). For exact array matching, use `expected`.
- **Nested objects** — subset matching applies recursively.

```typescript
// This passes:
// output:        { id: 'g2', description: 'Hello', tags: ['a', 'b', 'c'], meta: { views: 5, stars: 0 } }
// expectedMatch: { description: 'Hello', tags: ['a', 'b'], meta: { stars: 0 } }

// This fails:
// output:        { id: 'g2', description: 'Changed' }
// expectedMatch: { description: 'Hello' }  — 'Changed' !== 'Hello'
```

### `scenario()` for multi-step journey testing

A new testing primitive in `@ontrails/testing` for flows that span multiple trail invocations:

```typescript
import { scenario } from '@ontrails/testing';

scenario('Fork flow', app, [
  {
    cross: createGist,
    input: { description: 'Original', content: '# Hello' },
    as: 'original',
  },
  {
    cross: forkGist,
    input: { id: ref('original.id') },
    as: 'forked',
    expectedMatch: {
      content: '# Hello',
      forkedFrom: ref('original.id'),
    },
  },
]);
```

**Key design choices:**

- **`ref()` for cross-step references.** `ref('original.id')` reads the `id` field from the step aliased as `original`. This keeps scenarios as structured data — no callbacks, no lambdas, no imperative control flow. The reference is resolved at runtime when the step executes.
- **Each step is a cross.** The scenario runner invokes each trail through the normal execution pipeline — validation, layers, blaze, Result. It's not a shortcut. Layers run. Resources resolve. Tracing records.
- **`as` names the step's output.** Subsequent steps can reference it via `ref()`. If a step fails, the scenario stops and reports which step failed with the full Result.
- **`expectedMatch` on steps.** Each step can assert partial output, same semantics as on trail examples.
- **Scenarios are structured data.** They're arrays of objects with known shapes. The framework can read them the same way it reads examples — for agent guidance, documentation generation, warden analysis, and coverage reporting.

### Scenarios live in test files, not trail definitions

Trail examples live ON the trail — they're part of the contract, visible to agents, fed to six consumers. Scenarios live in test files alongside `testTrail` calls — they're multi-trail integration tests that verify composition behavior.

The distinction matters:

- **Examples** answer: "What does THIS trail do?" One trail, one input, one output.
- **Scenarios** answer: "How do these trails work TOGETHER?" Multiple trails, sequential steps, cross-step references.

Putting scenarios on a trail definition would mean one trail "owns" a multi-trail flow. That's backwards — the flow is a property of the composition, not of any single trail.

```typescript
// In a test file: gist.test.ts
import { scenario, testAll } from '@ontrails/testing';
import { app } from '../app';

// Examples test individual trails
testAll(app);

// Scenarios test composition flows
scenario('Create and star', app, [
  { cross: createGist, input: { description: 'Test', content: '# Hi' }, as: 'gist' },
  { cross: starGist, input: { gistId: ref('gist.id'), userId: 'u1' } },
  { cross: showGist, input: { id: ref('gist.id') }, expectedMatch: { starred: true } },
]);
```

### Warden integration

Two new suggestion-level rules:

| Rule | Severity | Description |
|---|---|---|
| `composite-example-coverage` | suggestion | Trail with `crosses` has no examples using `expectedMatch`. Suggests adding partial assertions. |
| `scenario-coverage` | suggestion | Crossing graph has connected components with no scenario tests. Suggests adding a scenario for the flow. |

These are coaching rules — suggestions, not errors. They guide the developer toward better coverage without blocking progress.

## Non-goals

- **Scenarios as a primitive.** `scenario()` is a testing utility in `@ontrails/testing`, not a framework primitive. It doesn't appear in the topo graph, the lockfile, or the trail spec. If usage patterns eventually justify promotion, that's a future decision.
- **Stateful scenarios.** Scenarios don't manage database state between steps. If a step creates a record, the next step sees it because they share the same test context and resources. The scenario runner doesn't provide setup/teardown hooks — use `beforeEach`/`afterEach` from the test runner for that.
- **Conditional branching in scenarios.** No `if/else` in scenario steps. If a flow has branches, write separate scenarios for each path. Scenarios are linear sequences, not control flow.
- **`expectedMatch` with custom matchers.** No `expectedMatch: { createdAt: expect.any(Date) }` or similar matcher integration. The assertion is structural equality on declared fields. For custom assertions, use `testTrail` with imperative checks.

## Consequences

### Positive

- **Composite trail examples become meaningful.** `expectedMatch` fills the gap between "assert everything" and "assert nothing." The developer says what matters and ignores what's unpredictable.
- **Journey tests are structured data.** `scenario()` keeps multi-step flows as declarative arrays, feeding the same consumers that examples feed — agent guidance, documentation, coverage analysis.
- **Progressive testing story.** Examples for individual trails. `expectedMatch` for composite trails. Scenarios for multi-trail flows. `testTrail` for edge cases. Each level adds precision without replacing the previous one.
- **No changes to existing testing.** `expected` and schema-only assertions continue to work exactly as today. `expectedMatch` and `scenario()` are additive.

### Tradeoffs

- **Two new concepts.** `expectedMatch` is a small addition to the example spec. `scenario()` is a new testing function. Both need documentation and examples of their own.
- **`ref()` is a mini-DSL.** Cross-step references introduce a resolution mechanism that doesn't exist elsewhere in examples. The tradeoff is accepted because the alternative — callbacks or lambdas — would break examples-as-data.
- **Scenario coverage is hard to measure.** The warden can detect that crossing graphs lack scenarios, but it can't assess whether existing scenarios cover the interesting paths. Coverage suggestions will be noisy initially.

## Non-decisions

- **`scenario()` step types beyond `cross`.** Steps that emit signals, wait for fires, or assert intermediate state. These expand the scenario vocabulary and should be designed when signal testing matures.
- **Scenario fixtures.** How scenarios declare and manage test data (seed records, deterministic IDs). This interacts with the broader fixtures design from the stash retro.
- **Snapshot-based `expectedMatch`.** Auto-generating `expectedMatch` from a passing run and storing it as a snapshot. Convenient but introduces snapshot maintenance burden. Deferred.
