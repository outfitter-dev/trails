# ADR-0008: Deterministic Trailhead Derivation

## Context

"Define once, trailhead everywhere" is the core promise. But it only works if derivation is deterministic — the same trail must produce the same CLI command, the same MCP tool name, and the same HTTP route every time, on every machine, without runtime state.

If derivation were heuristic or inference-based, developers and agents would have to run the framework to discover what a trail looks like on a given trailhead. That defeats the purpose. The derivation rules need to be simple enough that a human reading a trail definition can predict the CLI flag names, the MCP tool name, and the HTTP path without executing anything.

There are cases where the default derivation doesn't fit. A trail ID like `user.list` naturally derives to `GET /api/user/list`, but a team that wants RESTful resource paths would prefer `GET /api/users`. The derivation must be overridable per-trail without abandoning the defaults everywhere else.

Finally, derivation results need to be capturable as artifacts. Trailhead maps (generated by `@ontrails/schema`) serialize the full set of derivations for a topo into a diffable, hashable file. If a code change silently alters a derivation — renaming a trail, changing an intent, modifying a schema field — the trailhead map diff catches it. This closes the loop between "what the developer authored" and "what every trailhead actually produces."

## Decision

### CLI derivation

Trail IDs map to CLI commands via dot-splitting:

- The first dot separates group from command name: `entity.show` → group `entity`, name `show`
- Top-level IDs (no dot) have no group: `search` → name `search`
- `deriveFields(schema)` introspects the Zod input schema to produce a trailhead-agnostic `Field[]` descriptor array
- `toFlags(fields)` converts those descriptors to CLI flag definitions, including kebab-casing (`dryRun` → `--dry-run`)
- Destroy-intent trails auto-merge a `--dry-run` flag via `dryRunPreset()`

### MCP derivation

- `deriveToolName(appName, trailId)` produces an MCP-safe tool name: lowercase, dots and hyphens replaced with underscores, prefixed with the app name
- `deriveAnnotations(trail)` maps intent to MCP annotation hints via an explicit lookup table: `read` → `readOnlyHint: true`, `destroy` → `destructiveHint: true`
- `idempotent: true` on the trail produces `idempotentHint: true`
- `description` becomes `title`

### HTTP derivation

- `derivePath(basePath, trailId)` converts dots to path segments: `entity.show` → `/api/entity/show`
- `deriveMethod(trail)` maps intent to HTTP method via a three-entry lookup table: `read` → `GET`, `write` → `POST`, `destroy` → `DELETE`
- `deriveInputSource(method)` determines where input comes from: `GET` → query string, `POST` / `DELETE` → request body
- Route collisions (two trails deriving the same method + path) are detected at build time and returned as `Result.err(ValidationError)`

### Derivation properties

All derivation functions share four properties:

1. **Pure.** No side effects, no state, no I/O.
2. **Deterministic.** Same input always produces same output.
3. **Explicit lookup tables.** Intent-to-method, intent-to-hint, and field-type-to-flag mappings are all `Record` lookups, not conditional chains or pattern matching.
4. **Overridable.** Any derived value can be replaced via trail-level trailhead config. The override is the escape hatch; the derivation is the default.

### Example: `user.delete` with intent `destroy`

The following table shows how a single trail ID derives across all three trailheads for an app named `myapp` with base path `/api`:

| Trailhead | Derived artifact | Value |
| --- | --- | --- |
| CLI | Group | `user` |
| CLI | Command name | `delete` |
| CLI | Extra flags | `--dry-run` (auto-added for destroy intent) |
| MCP | Tool name | `myapp_user_delete` |
| MCP | Annotations | `destructiveHint: true` |
| HTTP | Method | `DELETE` |
| HTTP | Path | `/api/user/delete` |
| HTTP | Input source | `body` |

Every value in this table is computed from two inputs: the trail ID and the intent. No configuration file, no runtime discovery, no framework state. An agent reading the trail spec can produce this table without calling any function.

### Trailhead maps as drift detection

`generateSurfaceMap(topo)` captures every derivation for every trail in a topo as a serializable artifact. The output is diffable and hashable. When checked into source control or compared between CI runs, it answers the question: "did any trailhead projection change since last time?"

This is the governance complement to deterministic derivation. The derivation rules guarantee predictability. Trailhead maps guarantee visibility.

## Consequences

### Positive

- **Predictable projections.** Developers and agents can reason about what a trail looks like on any trailhead by reading the trail definition. No need to boot the framework.
- **Agent-native introspection.** An agent connecting to a Trails app can predict tool names, HTTP paths, and CLI commands from the contract alone.
- **Drift detection.** Trailhead maps make derivation changes visible in code review. A renamed trail ID shows up as a diff in every trailhead projection.
- **Zero-config defaults.** Most trails need no trailhead-specific configuration. The derivation rules produce sensible results for the common case.

### Tradeoffs

- **Action-oriented paths, not RESTful resource paths.** The default HTTP derivation produces paths like `/api/user/delete`, not `/api/users/:id` with a `DELETE` method. Teams that want REST-style paths need per-trail overrides. This is a deliberate choice: action-oriented paths are unambiguous and map cleanly from trail IDs. REST-style paths require additional information (pluralization, parameter extraction) that the trail contract doesn't encode.
- **Flat MCP tool names.** The underscore-delimited format (`myapp_user_delete`) loses the hierarchical structure of dot-separated trail IDs. MCP tool names are flat strings by spec. Hierarchy is recovered via description and annotation metadata, not the name itself.

### What this does NOT decide

- Whether a `rest` derivation mode will be added at the app or topo level to produce RESTful resource paths by default
- Whether trailhead-specific overrides will use a dedicated config block or inline trail metadata
- URL parameter extraction patterns for resource-oriented routes (e.g., `/users/:id`)
