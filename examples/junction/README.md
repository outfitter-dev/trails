# junction

Self-hosted webhook relay: receive, verify, store, re-deliver. One trail graph surfaces as an HTTP API (the hero), an MCP server, and an ops CLI.

## What this showcases

| Capability | Where it appears |
| --- | --- |
| Webhook routes with real signature verification | the [`webhook.inbound`](src/trails/webhook.ts) source declares `POST /hooks/:endpointId` with raw-body + allowlisted-header delivery; [`webhook.receive`](src/trails/webhook.ts) verifies GitHub `X-Hub-Signature-256`, Stripe `Stripe-Signature`, and generic HMAC-SHA256 over the raw body via [`verify.ts`](src/verify.ts) |
| Permits / JWT | `relay:read` / `relay:write` scopes declared per trail (e.g. [`endpoint.ts`](src/trails/endpoint.ts)); one JWT boundary for all three surfaces in [`permits.ts`](src/permits.ts); parity proven in [`permit-parity.test.ts`](__tests__/permit-parity.test.ts) |
| Error taxonomy | one class hierarchy → deterministic exit / HTTP / JSON-RPC codes (table below); ingress errors project through the same policy in [`server.ts`](src/server.ts) |
| OpenAPI | committed [`openapi.json`](openapi.json), regenerated by [`bin/generate-openapi.ts`](bin/generate-openapi.ts), served live at `GET /api/openapi.json`, drift-checked by [`openapi-artifact.test.ts`](__tests__/openapi-artifact.test.ts) |
| Hono adapter | management API derives through `@ontrails/hono` in [`server.ts`](src/server.ts) |
| Layers | [`pagination.ts`](src/layers/pagination.ts) owns `limit`/`offset` on `event.list` and `delivery.list` — projected as HTTP query params, CLI flags, and MCP tool params |
| Signals | [`event.received`](src/signals.ts) fired by `webhook.receive`/`event.replay`, consumed by [`relay.dispatch`](src/trails/relay.ts) via `on:` |
| Detours (cameo) | [`delivery.send`](src/trails/delivery.ts) retries network failures 3× with backoff, then records the failure (`lookout` owns the deep detour story) |
| Store | five entities in [`store.ts`](src/store.ts) bound to SQLite through `@ontrails/drizzle` with a deterministic in-memory mock ([`relay-store.ts`](src/resources/relay-store.ts)) |
| Offline testing | `testAll(graph)` plus the end-to-end relay test run entirely against mocked resources |

## Quickstart

From the repo root, `bun install`, then everything below runs from `examples/junction/`.

**Terminal 1 — start junction:**

```bash
bun bin/serve.ts
```

**Terminal 2 — start a local delivery target:**

```bash
bun bin/echo-target.ts
```

**Terminal 3 — register an endpoint, target, and route, then send a signed webhook** (uses `jq` to capture ids):

```bash
export TOKEN=$(bun bin/mint-token.ts admin)

ENDPOINT=$(bun bin/junction.ts endpoint create --name "CI hooks" --source github --token "$TOKEN" --json)
export EP=$(echo "$ENDPOINT" | jq -r .id) SECRET=$(echo "$ENDPOINT" | jq -r .secret)
# the secret is shown exactly once — list/get project it redacted

export TGT=$(bun bin/junction.ts target create --name "Local echo" --url http://localhost:9999/relay --token "$TOKEN" --json | jq -r .id)

bun bin/junction.ts route create --endpoint-id "$EP" --target-id "$TGT" --token "$TOKEN" --json

# sign a payload the way GitHub would and deliver it
bun bin/send-webhook.ts "$EP" github "$SECRET" '{"zen":"keep it logically awesome"}'
```

Terminal 1 accepts the request (`202` with `{"data":{…,"signatureValid":true}}`), and Terminal 2 prints the relayed payload: `received POST {"zen":"keep it logically awesome"}`.

**Watch the relay, then view the same event over MCP with a viewer token:**

```bash
bun bin/junction.ts event list --token $(bun bin/mint-token.ts viewer) --json

JUNCTION_MCP_TOKEN=$(bun bin/mint-token.ts viewer) bun bin/mcp-call.ts junction_event_list '{}'
```

A tampered signature is rejected and still recorded for audit:

```bash
bun bin/send-webhook.ts "$EP" github wrong-secret '{"x":1}'
# 401 {"error":{"category":"auth","code":"AuthError","message":"Signature verification failed ..."}}
bun bin/junction.ts event list --status dead --token "$TOKEN" --json
```

To connect a real MCP client, launch `bun src/mcp.ts` over stdio with `JUNCTION_MCP_TOKEN` set; tools carry the same permits as HTTP and CLI.

## One declaration, three enforcements

Every mutation trail declares `permit: { scopes: ['relay:write'] }` once. That single declaration rejects a viewer JWT on:

- **HTTP** — `POST /api/endpoint/disable` with a viewer bearer token → `403` (`PermitError`, category `permission`)
- **MCP** — `junction_endpoint_disable` with a viewer token → `isError` with the same `PermitError` projection
- **CLI** — `bun bin/junction.ts endpoint disable --token <viewer-jwt>` → exit code `4`

[`permit-parity.test.ts`](__tests__/permit-parity.test.ts) proves the HTTP and MCP halves with real HS256 JWTs; nothing is stubbed between the bearer token and the permit check.

## Error taxonomy on the wire

One error class hierarchy maps deterministically onto every surface. The rows junction exercises:

| Error class | Category | CLI exit | HTTP status | JSON-RPC code |
| --- | --- | --- | --- | --- |
| `AuthError` (bad webhook signature) | auth | 9 | 401 | -32600 |
| `PermitError` (viewer attempting a mutation) | permission | 4 | 403 | -32600 |
| `NotFoundError` (unknown endpoint/event/target/route) | not_found | 2 | 404 | -32601 |
| `ValidationError` (malformed payload, disabled target) | validation | 1 | 400 | -32602 |
| `NetworkError` (target unreachable, pre-detour) | network | 7 | 502 | -32603 |
| `RetryExhaustedError` (delivery detour exhausted) | inherits wrapped `NetworkError` → network | 7 | 502 | -32603 |

## Framework gaps found while building (dogfood payoff)

Building the first cut of this app surfaced three framework gaps, filed as Linear issues and since fixed upstream — the app now uses the real APIs:

1. **TRL-1174 (fixed by webhook ingress v2)** — `webhook()` sources now support dynamic path segments, so `POST /hooks/:endpointId` is declared on the [`webhook.inbound`](src/trails/webhook.ts) source instead of a hand-authored route.
2. **TRL-1175 (fixed by webhook ingress v2)** — sources can opt into `rawBody` delivery and an allowlisted `headers` envelope, so `webhook.receive` gets the exact signed bytes through the primitive. Verification stays in the blaze deliberately: a rejected signature must still record its audit event before the 401.
3. **TRL-1176 (fixed)** — `createMcpHarness` forwards the full `McpExtra`, so the permit-parity suite drives bearer tokens straight through the harness.
